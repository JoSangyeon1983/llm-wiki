# LLM Wiki — Schema & Operating Manual

Andrej Karpathy의 **LLM Wiki** 패턴을 따르는 개인 지식 저장소.
원본을 질문할 때마다 다시 읽는 RAG 방식 대신, **수집(ingest) 시점에 지식을 위키로
컴파일**하고 시간이 지날수록 복리(compounding)로 축적함. LLM(Claude)이 마크다운
문서를 직접 작성·편집·연결하는 **위키 에디터** 역할을 함.

이 파일은 LLM이 위키를 **수집/유지/질의**하는 방법을 정의하는 스키마임.
새 세션을 시작할 때 가장 먼저 이 파일을 읽음.

---

## 1. 핵심 원칙

1. **3-레이어 분리** — 불변 원본(`sources/`) · LLM 소유 위키(PARA 폴더) · 스키마(이 파일).
2. **LLM이 위키 에디터** — 사용자가 주제/소스를 던지면 Claude가 문서를 생성·갱신함.
3. **복리 축적** — 한 번에 완벽히 쓰지 않고, 수집·질의를 거듭하며 보강함.
4. **출처 추적(provenance)** — 모든 주장은 `sources/`의 source-id로 역추적 가능해야 함.
5. **Atomic Notes** — 한 파일에 한 주제. 파일명이 곧 제목.
6. **한국어 우선** — 문서는 한국어로 작성. 기술 용어는 영문 병기 가능.

## 2. 3-레이어 아키텍처

| 레이어 | 위치 | 소유자 | 변경 규칙 |
|--------|------|--------|-----------|
| **Raw sources** | `sources/` | 사람 | 불변(immutable). LLM은 **읽기만** 함. |
| **Wiki** | PARA 6폴더 (`00_~05_`) | LLM | LLM이 생성·갱신·연결함. |
| **Schema** | `CLAUDE.md` (이 파일) | 사람+LLM | 규약을 정의함. |

> `sources/`(원본·불변)와 `03_References/`(LLM이 쓴 요약·인용 페이지)는 다름.
> 원본은 `sources/`에 그대로 보존하고, 그 가공물만 위키 레이어에 둠.

## 3. 디렉토리 구조

```
/                     # vault root
├── CLAUDE.md         # 이 파일 — 스키마/작업 지침
├── sources/          # ★ 원본(불변) — 논문·기사·데이터·메모 + source-id 레지스트리
│   └── README.md     # 소스 등록 규칙, source-id 인덱스 표
├── 00_Index/         # 목차·지도 문서 (MOC: Map of Content) + answer 환원 페이지 등록
├── 01_Concepts/      # 개념 정리 (정의, 원리, 이론)
├── 02_HowTo/         # 실용 가이드, 레시피, 튜토리얼
├── 03_References/    # 논문 요약, 인용, 외부 링크 정리 (sources/의 가공물)
├── 04_Projects/      # 프로젝트별 노트
├── 05_Logs/          # log.md (append-only) + 학습 일지·실험 기록
└── assets/           # 이미지, 첨부파일
```

## 4. 문서 작성 규칙

> **작성 전 반드시 [`WRITING_GUIDE.md`](WRITING_GUIDE.md)를 읽음.** 문서를 새로 쓰거나 고칠 때
> 적용할 **문체(두괄식·개조식·음슴체)와 문장·표현 품질**(쉽게·읽는 이 처지에서·정확하게 쓰기 + 작성 3단계 + 점검표)을
> 정의함.
>
> **역할 분리** — 이 §4는 **파일·구조·메타데이터·링크·출처 메커니즘** 규약만 다룸.
> **"쓰는 법"(문체·문장·표현)의 정본은 WRITING_GUIDE.md**임.

### 파일명
- 공백 허용 (Obsidian 호환). 한국어 파일명 가능 (예: `트랜스포머.md`).
- 영문 약어가 보편적이면 영문 사용 (예: `LLM.md`, `RAG.md`).

### 프론트매터 (YAML)
모든 문서 상단에 포함:

```yaml
---
title: 문서 제목
type: concept | howto | reference | project | log | answer
status: stub | draft | stable
tags: [태그1, 태그2]
sources: [source-id, ...]        # sources/ 의 항목 id (해당 시)
created: YYYY-MM-DD
updated: YYYY-MM-DD
---
```

- `type: answer` — 질의로부터 위키에 **환원된** 페이지.
- `status` — `stub`(껍데기) → `draft`(초안) → `stable`(안정).

### 본문 구조
```markdown
# 제목

> 한 줄 요약 (TL;DR)

## 핵심 내용
- ...

## 세부 설명
- ...

## 관련 문서
- [[관련문서1]]
- [[관련문서2]]

## 참고
- 출처는 본문에 (출처: source-id)로 표기. 여기엔 보조 링크/URL.
```

### 문체
- 두괄식·개조식·음슴체·간결 — **정의는 [`WRITING_GUIDE.md`](WRITING_GUIDE.md) §2(문체)**를 따름. 여기선 재정의하지 않음.

### Wikilink 규칙
- 새 개념이 등장하면 `[[개념명]]`으로 링크함.
- 링크 대상 문서가 아직 없어도 괜찮음 — 다음에 만들 stub 신호.
- 별칭은 `[[파일명|표시텍스트]]` 형식.

### 출처·인용 규칙
- 주장에는 출처를 명시함: `... (출처: source-id)`.
- 원본을 인용·요약한 페이지는 frontmatter `sources:`에 해당 id를 나열함.

### 모순(contradiction) 처리
- 기존 지식과 충돌하는 정보는 **삭제하지 말고** 블록으로 보존함:

```markdown
> ⚠️ 모순: A는 X라 함 (출처: src-1) / B는 ¬X라 함 (출처: src-2). 미해결.
```

## 5. 수집 워크플로 (Ingest)

새 소스를 `sources/`에 추가하고 `sources/README.md` 표에 등록한 뒤:

1. 소스를 읽고 핵심 엔티티/개념/주장을 추출함.
2. 관련된 기존 페이지를 **모두** 갱신함 (한 소스가 여러 페이지를 연쇄 변경).
   없으면 적절한 PARA 폴더에 새 페이지(처음엔 `status: stub` 가능)를 만듦.
3. 새/변경 페이지를 `[[ ]]`로 상호 연결하고 출처를 표기함.
4. 기존 지식과 충돌하면 모순 블록으로 보존함.
5. `00_Index/`의 MOC에 신규 페이지를 등록함.
6. `05_Logs/log.md`에 한 줄 기록을 append 함.

## 6. 질의 워크플로 (Query)

1. `00_Index/` MOC에서 관련 페이지를 찾음.
2. 해당 페이지(및 `[[링크]]` 이웃)를 읽고 **인용과 함께** 답을 합성함.
3. 위키에 없던 새 합성·탐색 결과는 `type: answer` 페이지로 **환원**해
   적절한 폴더에 두고 MOC·log에 등록함 (탐색이 휘발되지 않게 함).

## 7. 유지보수 규칙 (Lint)

- MOC(00_Index)와 실제 페이지는 항상 일치시킴 (추가/삭제 시 동시 갱신).
- 끊어진 `[[링크]]`, 고아 페이지, 오래된 `stub`은 다음 수집 때 우선 보강함.
- `updated` 날짜를 갱신함.

## 8. 로그 (`05_Logs/log.md`)

**append-only.** 최신 항목을 **아래**에 추가함. 과거 줄은 수정·삭제하지 않음.
한 줄 포맷 (grep/Unix 도구 파싱 친화적):

```
YYYY-MM-DD | <작업유형> | <대상> | <한 줄 설명>
```

**작업유형**
- `INGEST` — 새 소스를 읽고 위키에 통합
- `QUERY` — 위키에 질문하고 답변 생성 (환원 포함 시 명시)
- `CREATE` — 새 문서 생성
- `UPDATE` — 기존 문서 수정
- `LINT` — 위키 상태 점검 (모순·고아·누락 링크)
- `MAINT` — 구조·스키마 등 기타 정비

규칙: 작업이 끝날 때마다 반드시 기록함. log.md가 없으면 먼저 생성함.

---

## 9. 하지 말 것

- 사용자가 요청하지 않은 문서를 임의로 삭제하지 않음.
- `sources/`의 원본을 수정하지 않음 (읽기 전용).
- 확실하지 않은 정보를 사실처럼 적지 않음 — 불확실하면 명시하거나 모순 블록 사용.
- 출처 없는 주장을 단정하지 않음.
- 영어로 본문을 작성하지 않음 (기술 용어 병기는 허용).
- `assets/` 외 위치에 바이너리 파일을 두지 않음.
- log.md 기록을 생략하지 않음. 과거 로그를 고쳐 쓰지 않음.
