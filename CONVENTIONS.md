> 이 위키의 디렉토리 구조, 문서 작성 규칙, 프론트매터, 문체, 인용 원칙, 로그 양식을 정의하는 작성 표준임. AI 운영 규칙(수집/질의/검수/유지 워크플로)은 [[AGENTS|에이전트 작업 지침]]에서 다룸. AI/사람 누구든 위키에 글을 쓸 때 따름. CLAUDE.md → AGENTS.md → CONVENTIONS.md(이 파일)로 import 됨.

# 1. 핵심 원칙

Andrej Karpathy의 **LLM Wiki** 패턴을 따르는 개인 지식 저장소. 원본을 질문마다 다시 읽는 RAG 대신, **수집(ingest) 시점에 지식을 위키로 컴파일**하고 시간이 지날수록 복리(compounding)로 축적함. LLM(Claude)이 마크다운 문서를 직접 작성·편집·연결하는 위키 에디터 역할을 함([[AGENTS]] §1).

1. **3-레이어 분리** — 불변 원본(`sources/`) · LLM 소유 위키(PARA 폴더) · 스키마(`CLAUDE.md`/`AGENTS.md`/`CONVENTIONS.md`).
2. **Atomic Notes** — 한 파일에 한 주제. 파일명이 곧 제목.
3. **복리 축적** — 한 번에 완벽히 쓰지 않고, 수집·질의를 거듭하며 보강함.
4. **출처 추적(provenance)** — 모든 주장은 `sources/`의 source-id로 역추적 가능해야 함.
5. **엔티티 1급** — 고유명사(사람·조직·모델·제품·논문)는 `type: entity` 페이지로 분리해 상호연결의 허브로 삼음.
6. **링크 하한선** — 모든 페이지는 관련 페이지를 **최소 2개** `[[ ]]`로 연결함(고아 방지·복리 그래프 강제). 순수 색인(MOC)·로그는 예외.
7. **한국어 우선** — 문서는 한국어로 작성. 기술 용어는 영문 병기 가능.

# 2. 3-레이어 아키텍처

| 레이어 | 위치 | 소유자 | 변경 규칙 |
|--------|------|--------|-----------|
| **Raw sources** | `sources/` | 사람 | 불변(immutable). LLM은 **읽기만** 함. |
| **Wiki** | PARA 6폴더 (`00_~05_`) | LLM | LLM이 생성·갱신·연결함. |
| **Schema** | `CLAUDE.md` + `AGENTS.md` + `CONVENTIONS.md` | 사람+LLM | 규약을 정의함. |

> `sources/`(원본·불변)와 `03_References/`(LLM이 쓴 요약·인용 페이지)는 다름.
> 원본은 `sources/`에 그대로 보존하고, 그 가공물만 위키 레이어에 둠.

# 3. 디렉토리 구조

```
/                     # vault root
├── CLAUDE.md         # Claude Code 자동 로드 진입점 (AGENTS.md import 스텁)
├── AGENTS.md         # 에이전트 작업 지침 (AI 운영 규칙)
├── CONVENTIONS.md    # 이 파일 — 위키 작성 규칙
├── sources/          # ★ 원본(불변) — 논문·기사·데이터·메모 + source-id 레지스트리
│   └── README.md     # 소스 등록 규칙, source-id 인덱스 표
├── 00_Index/         # 목차·지도 문서 (MOC: Map of Content) + answer 환원 페이지 등록
├── 01_Concepts/      # 개념 정리 (정의·원리·이론) + 엔티티(type:entity) + 종합(type:overview)
├── 02_HowTo/         # 실용 가이드, 레시피, 튜토리얼
├── 03_References/    # 논문 요약, 인용, 외부 링크 정리 (sources/의 가공물)
├── 04_Projects/      # 프로젝트별 노트
├── 05_Logs/          # log.md (append-only) + 학습 일지·실험 기록
├── scripts/          # 자동화 스크립트 (wiki_lint.mjs 등)
└── assets/           # 이미지, 첨부파일
```

# 4. 문서 작성 규칙

## 파일명
- 공백 허용 (Obsidian 호환). 한국어 파일명 가능 (예: `트랜스포머.md`).
- 영문 약어가 보편적이면 영문 사용 (예: `LLM.md`, `RAG.md`).

## 프론트매터 (YAML)
모든 위키 문서 상단에 포함:

```yaml
---
title: 문서 제목
type: entity | concept | overview | howto | reference | project | log | answer | index
status: stub | draft | stable
tags: [태그1, 태그2]
sources: [source-id, ...]        # sources/ 의 항목 id (해당 시)
created: YYYY-MM-DD
updated: YYYY-MM-DD
---
```

- `type: entity` — 고유명사(사람·조직·모델·제품·논문 등) 페이지. **위키 상호연결의 중추.**
  PARA 폴더는 유지하되 엔티티는 주로 `01_Concepts/`에 두고 `type`로 구분함.
- `type: concept` — 개념·원리·이론의 정의 페이지. 검수 시 정의 정확성·상위/하위 개념 혼동을 봄([[AGENTS]] 검수 B블록).
- `type: overview` — 한 주제를 **서술형으로 종합**한 페이지(목차인 MOC와 다름). §5 참고.
- `type: index` — MOC 등 색인 페이지.
- `type: answer` — 질의로부터 위키에 **환원된** 페이지.
- `status` — `stub`(껍데기) → `draft`(초안) → `stable`(안정).

> 예외: 폴더 placeholder인 각 `README.md`와 `05_Logs/log.md`는 위키 콘텐츠 페이지가 아닌 인프라 문서라 프론트매터·본문구조 규칙을 면제함. 스키마 파일(`CLAUDE.md`·`AGENTS.md`·이 파일)은 메타 문서로 별도 취급함.

## 본문 구조
- **H1 제목 라인을 두지 않음** — frontmatter `title`이 유일한 제목원. 본문에 `# 문서제목`을 다시 쓰지 않음(중복 방지).
- 본문은 한 줄 요약(TL;DR) 인용블록부터 시작하고, 섹션은 H1(`#`)부터 사용함.
- 예외: `05_Logs/log.md`는 `# Log` H1을 컨테이너 헤더로 유지하고, 폴더 `README.md`도 기존 H1을 유지함.

```markdown
> 한 줄 요약 (TL;DR)

# 1. 핵심 내용
- ...

# 2. 세부 설명
- ...

# 3. 관련 문서
- [[관련문서1]]
- [[관련문서2]]

# 4. 참고
- 출처는 본문에 (출처: source-id)로 표기. 여기엔 보조 링크/URL.
```

## 문체
- **두괄식** — 결론·핵심을 먼저, 부연은 뒤에.
- **개조식** — 불릿 중심으로 간결하게.
- **음슴체** — "~이다/~한다" 대신 "~임/~함" 체.
- 한 문장은 짧게, 불필요한 수식어 제거. 표·리스트를 적극 활용.

## 표현 형식 선택 기준
내용 성격에 맞는 구조를 고름 (작성·검수 공통 적용). 검수는 [[AGENTS]] 검수 형식(A) 블록에서 점검함.

| 형식 | 사용 기준 |
|------|-----------|
| **도표(표)** | 비교가 필요할 때 · 같은 기준으로 여러 항목을 정리할 때 · 조건과 결과를 연결할 때 · 상태값·데이터 필드·리소스 목록·예외 처리를 정의할 때 |
| **1단계 블릿** | 짧은 속성을 정리할 때 · "라벨: 내용"으로 끝나는 정보일 때 · 항목들이 서로 동등한 수준일 때 · 빠른 확인이 목적일 때 · 단순 목록·요약을 제시할 때 |
| **서브 블릿** | 상위 항목을 세부 설명할 때 · 조건·예외·예시·절차가 필요할 때 · 담당자별 작업이 갈릴 때 · 입력값·출력값·저장값을 함께 설명할 때 · 한 줄로 쓰면 너무 길어지는 규칙을 정리할 때 |
| **번호 목록** | 순서가 중요한 절차·플로우를 설명할 때 |

## Wikilink 규칙
- **모든 페이지는 관련 페이지를 최소 2개 `[[ ]]`로 링크함** (예외: 순수 색인/로그/README/스키마 파일). 링크 부족은 lint 대상.
- 새 개념·고유명사가 등장하면 `[[개념명]]`으로 링크함.
- 링크 대상 문서가 아직 없어도 괜찮음 — 다음에 만들 stub 신호.
- 별칭은 `[[파일명|표시텍스트]]` 형식.
- 섹션 anchor는 `[[파일명#섹션 헤딩 텍스트]]` 형식 (헤딩 텍스트 전체를 그대로 씀, 예: `[[CONVENTIONS#4. 문서 작성 규칙]]`). 블록 임베드는 `![[파일명#^블록ID]]`. lint가 anchor·블록ID 정합을 검증함.

## 출처·인용 규칙
- 주장에는 출처를 명시함: `... (출처: source-id)`.
- 원본을 인용·요약한 페이지는 frontmatter `sources:`에 해당 id를 나열함.

## 모순(contradiction) 처리
- 기존 지식과 충돌하는 정보는 **삭제하지 말고** 블록으로 보존함:

```markdown
> ⚠️ 모순: A는 X라 함 (출처: src-1) / B는 ¬X라 함 (출처: src-2). 미해결.
```

# 5. Overview(종합) 페이지

- **정의**: 한 주제의 현재 이해를 **서술형으로 합성**한 페이지(`type: overview`). 단순 링크 목록인 MOC와 구분됨.
- **역할**: 흩어진 엔티티·개념 페이지를 하나의 이야기로 엮어 "지금까지 아는 것"을 보여줌.
- **위치**: 주제가 속한 PARA 폴더(주로 `01_Concepts/`)에 둠. 파일명 예: `LLM 개요.md`.
- **유지**: 관련 엔티티/개념이 갱신되면 함께 갱신함. 본문에서 구성 페이지를 `[[ ]]`로 링크하고 MOC에 등록함.
- **MOC와의 관계**: MOC(00_Index)는 "어디에 무엇이 있는가"(색인), Overview는 "그래서 무엇을 아는가"(종합).

# 6. 로그 항목 양식

`05_Logs/log.md`에 추가되는 항목의 양식을 정의함. 기록 의무와 시점은 [[AGENTS]] §6에서 다룸.

- **append-only.** 최신 항목을 **위**에 추가함. 과거 줄은 수정·삭제하지 않음.
- `# Log` H1 컨테이너 아래 항목을 H2 블록으로 진행함 (Unix 도구 파싱 친화적).

```markdown
## [YYYY-MM-DD] {작업유형} | {제목 또는 주제}

- 작업 내용 요약
- 생성/수정된 파일 목록
- 새로 생성된 wikilink
```

**작업유형**
- `INGEST` — 새 소스를 읽고 위키에 통합
- `QUERY` — 위키에 질문하고 답변 생성 (환원 포함 시 명시)
- `CREATE` — 새 문서 생성
- `UPDATE` — 기존 문서 수정
- `REVIEW` — Codex 교차검수 결과 반영 ([[AGENTS]] §4)
- `LINT` — 위키 상태 점검 (모순·고아·누락 링크)
- `MAINT` — 구조·스키마 등 기타 정비
